# Ship（发布）技能

## 核心原则

**每次发布都应该：可回滚、可观测、增量推进。**

发布不只是"把代码推上去"——是把代码安全推上去，监控在位、回滚方案就绪、明确知道成功长什么样。

## 预发布检查清单

### 代码质量
- [ ] 所有测试通过（单元、集成、e2e）
- [ ] 构建成功，无 warning
- [ ] Lint 和类型检查通过
- [ ] 代码已 review 并 approved
- [ ] 无遗留 TODO（应在发布前解决的）
- [ ] 生产代码中无 console.log 调试语句
- [ ] 错误处理覆盖预期的失败场景

### 安全
- [ ] 无 secret 提交到版本控制
- [ ] 依赖安全审计无 critical/high 漏洞
- [ ] 所有用户输入端点有输入校验
- [ ] 认证和鉴权检查到位
- [ ] CORS 配置为具体域名（不用 *）
- [ ] 认证端点有速率限制

### 性能
- [ ] 核心链路无 N+1 查询
- [ ] 数据库查询有合适索引
- [ ] 列表接口有分页
- [ ] 静态资源配置缓存

### 基础设施
- [ ] 生产环境变量已配置
- [ ] 数据库迁移已应用（或已准备好应用）
- [ ] 日志和错误上报已配置
- [ ] 健康检查接口存在且正常响应

## 分级发布策略

```
1. 部署到 staging（feature flag 关闭）
   └── 完整测试套件在 staging 环境运行
   └── 关键流程手工冒烟测试

2. 部署到 production（feature flag 关闭）
   └── 验证部署成功（健康检查）
   └── 检查错误监控（无新错误）

3. 团队内开启（feature flag 对内部用户开启）
   └── 团队在生产环境使用功能
   └── 24 小时监控窗口

4. 灰度发布 5% 用户
   └── 监控：错误率、延迟、业务指标
   └── 24-48 小时观察窗口
   └── 满足以下阈值才推进：
       - 错误率：≤ 基线 10%（绿）/ 10-100%（黄）/ >2x（回滚）
       - P95 延迟：≤ 基线 20%（绿）/ 20-50%（黄）/ >50%（回滚）

5. 逐步扩大（25% → 50% → 100%）
   └── 每步监控相同指标
   └── 随时可回滚到上一个百分比

6. 全量发布
   └── 监控 1 周
   └── 清理 feature flag
```

## 回滚方案模板

每次发布前必须准备，不是事后补救：

```markdown
## [功能/版本] 回滚方案

### 触发条件
- 错误率 > 基线 2 倍
- P95 延迟 > [X]ms
- 用户反馈中特定问题激增
- 数据完整性异常

### 回滚步骤
1. 关闭 feature flag（如适用）—— < 1 分钟
   OR
1. 部署上一个版本：`git revert <commit> && push`—— < 5 分钟
2. 验证回滚：健康检查 + 错误监控
3. 通知团队

### 数据库考量
- 迁移 [X] 是否有回滚脚本？[是/否]
- 新功能写入的数据：[保留/清理]
```

## 发布后的第一小时

```
1. 检查健康接口返回 200
2. 检查错误监控面板（无新错误类型）
3. 检查延迟面板（无回归）
4. 手工测试关键用户流程
5. 确认日志正常输出
6. 确认回滚机制可用
```

## Red Flags

- 没有回滚方案就发布
- 生产环境无监控或错误上报
- 一次性大爆炸发布（没有分级，没有 staging）
- feature flag 没有过期时间或负责人
- 发布后前 1 小时无人监控
- 周五下午发布

## Verification

发布前：
- [ ] 预发布检查清单全部通过
- [ ] feature flag 已配置（如适用）
- [ ] 回滚方案已记录
- [ ] 监控面板已配置
- [ ] 团队已通知

发布后：
- [ ] 健康检查返回 200
- [ ] 错误率正常
- [ ] 延迟正常
- [ ] 关键用户流程可用
- [ ] 日志正常输出
