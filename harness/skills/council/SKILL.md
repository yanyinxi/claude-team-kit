# council — 四声部决策 Skill

## 核心价值

**不做单一答案机器**，而是提供多角度分析，让用户做出更明智的决策。

```
决策请求
    ↓
┌─────────────────────────────────────────┐
│  Council — 4 Voices                      │
│                                         │
│  🏛 Architect  架构正确性                │
│  🎭 Skeptic    前提质疑                  │
│  ⚡ Pragmatist 交付速度                 │
│  🔍 Critic     失败模式                  │
└─────────────────────────────────────────┘
    ↓
共识 + 最强异议 + 前提检查
```

## 4个声音详解

### 🏛 Architect（建筑师）

**职责**：长期正确性、技术债务、架构一致性

**分析维度**：
- 架构原则（SOLID、DRY、内聚）
- 技术债务影响
- 可扩展性
- 可测试性

**声音示例**：
```
🏛 Architect：

  方案 A（微服务）的问题：
  • 引入服务间通信复杂度
  • 增加运维负担（k8s/istio）
  • 当前团队规模（3人）ROI 为负

  方案 B（模块化单应用）：
  • 保持部署简单
  • 延迟复杂度，等业务验证
  • 更符合当前阶段

  建议：选择 B（渐进式演进），避免过度工程
```

---

### 🎭 Skeptic（怀疑者）

**职责**：质疑前提、挑战假设、识别盲点

**分析维度**：
- 用户陈述的假设是否成立？
- 是否有反例？
- 是否有被忽视的约束？
- "这个问题真的是这个问题吗？"

**声音示例**：
```
🎭 Skeptic：

  前提检验：
  • "需要支持 10 万并发" → 当前 DAU 多少？测量过吗？
  • "必须用微服务" → 为什么？单应用无法满足吗？
  • "性能是关键瓶颈" → Profiler 数据在哪里？

  建议：在投入方案前，先验证这些前提是否真实存在
```

---

### ⚡ Pragmatist（务实者）

**职责**：快速交付、最小可行方案、风险可控

**分析维度**：
- 多久能交付？
- 最小可行方案是什么？
- 风险点在哪里？
- 如何降低风险？

**声音示例**：
```
⚡ Pragmatist：

  务实建议：

  第一步（1天）：直接在现有单应用中新增 /api/v2/users
    • 零额外基础设施
    • 快速验证需求

  第二步（1周）：如果 v2 用户增长到 1000+
    • 再评估是否需要拆分

  避免：为了"未来扩展"多花 3 周设计微服务
```

---

### 🔍 Critic（批评者）

**职责**：识别失败模式、规划灾难场景、找出漏洞

**分析维度**：
- 最坏情况会发生什么？
- 如果这个库/服务停维护呢？
- 数据迁移风险？
- 安全性隐患？

**声音示例**：
```
🔍 Critic：

  失败模式分析：

  方案 A 的风险：
  • Postgres 作为主存储 → 单点故障
    → 缓解：增加 read replica
  • Redis 缓存失效 → 击穿数据库
    → 缓解：实现 cache-aside 模式

  方案 B 的风险：
  • DynamoDB 成本不可预测
    → 缓解：设置 spending alert
  • vendor lock-in
    → 缓解：使用 abstraction layer

  最强异议：缺少 DynamoDB 经验的团队遇到问题时响应慢
```

## 使用场景

### 场景 1：技术选型

```
> Council：我们的用户服务应该用 Postgres 还是 DynamoDB？

（4 个声音分别分析，输出推荐）
```

### 场景 2：架构重构

```
> Council：我们是否应该将单应用拆分为微服务？
```

### 场景 3：代码审查

```
> Council review：这段代码是否满足安全要求？
```

### 场景 4：方案评审

```
> Council：方案 A vs 方案 B，哪个更好？
```

## 决策规则

**何时采信最强异议**：
- 异议者能提供数据支持
- 异议涉及安全性或合规性
- 异议涉及可逆性低的决策

**何时忽略异议**：
- 异议仅基于偏好而非数据
- 异议可以通过分阶段交付缓解
- 时间紧迫且风险可控

## 验证方法

```bash
[[ -f skills/council/SKILL.md ]] && echo "✅"

for voice in "Architect" "Skeptic" "Pragmatist" "Critic"; do
  grep -qi "$voice" skills/council/SKILL.md && echo "✅ $voice" || echo "❌ $voice"
done

grep -q "最强异议\|一致决策\|前提检查" skills/council/SKILL.md && echo "✅ 输出格式"
```

## Red Flags

- 只有单一声音（Pragmatist 说了算）
- Skeptic 从不质疑前提
- Architect 给出的是意见而非架构原则
- 最强异议被忽略且无合理解释
